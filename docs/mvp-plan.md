# 回音笺 MVP 规划

## MVP 目标

第一阶段的目标不是构建完整产品，而是验证一个最关键的问题：

用户输入一段关于自己近期行动的自然语言描述后，系统能否稳定输出一份有价值、可读、可展示的“可见回报”结构化结果。

如果这条链路成立，就说明回音笺的核心体验具备产品化基础。

## 本阶段要做的最小链路

本阶段建议只实现以下最小链路：

1. 用户提交一段文本描述。
2. 后端接收并校验输入。
3. 调用模型或占位逻辑生成结构化反馈。
4. 返回统一的 JSON 输出结果。
5. 前端暂时只保留目录结构，不进入页面开发。

这一阶段重点验证的是“输入是否能被有效转译为可见回报”，而不是完整的账户体系、数据存储、页面交互或复杂运营机制。

## 暂时不做的内容

为了保证 MVP 聚焦，本阶段不建议投入以下内容：

- 用户登录与权限体系
- 数据库存储
- 多轮对话能力
- 复杂提示词编排系统
- 语音识别与音频上传
- 历史记录管理
- 数据分析面板
- 完整前端界面
- 多模型切换与路由

这些内容都可能在后续阶段增加，但不应影响第一版核心链路的验证。

## 建议技术结构

### 后端

- 使用 Python 作为主要开发语言
- 使用 FastAPI 作为 API 框架
- 使用 Pydantic 定义输入输出模型
- 使用 `requirements.txt` 管理依赖

### 模型调用

第一阶段可以先保留占位服务层，后续在 `services/` 中接入实际模型调用逻辑。建议模型层职责包括：

- 接收原始输入
- 应用统一 system prompt
- 约束模型按固定 JSON 输出
- 对输出做基础校验与兜底处理

### 目录建议

- `backend/app/main.py`：FastAPI 应用入口
- `backend/app/schemas.py`：请求与响应数据模型
- `backend/app/services/`：模型调用与结构化解析逻辑
- `docs/`：产品、MVP 和 Prompt 文档

## 开发步骤

### 第一步：仓库初始化

建立基础目录、依赖文件、说明文档和最小应用入口，确保项目可以启动。

### 第二步：定义统一输出结构

先确定输入输出 schema，尤其是“任务、产出、能力、简历表达、回报评分、下一步建议”这些核心字段。

### 第三步：补充分析接口

新增一个最小文本分析接口，先使用占位逻辑或模拟返回，打通调用链路。

### 第四步：接入模型

将 Prompt 文档中的 system prompt 应用于实际模型调用，并确保输出尽量稳定。

### 第五步：人工验证

准备若干真实输入样本，检查返回结果是否真实、保守、清晰，并评估其对用户是否有正反馈价值。

## 验证标准

MVP 可认为通过初步验证，需要至少满足以下标准：

- 服务可以本地正常启动
- 接口输入输出格式稳定
- 模型输出字段完整，便于前端直接消费
- 输出内容不过度夸大，不虚构结果
- 用户能够从结果中看见“我做了什么”和“这件事为什么有价值”
- 至少能为一部分输入生成可用的简历表达或下一步建议

如果以上标准能够稳定满足，就可以进入下一阶段：完善分析接口、增加示例数据、逐步接入前端页面与语音输入。
